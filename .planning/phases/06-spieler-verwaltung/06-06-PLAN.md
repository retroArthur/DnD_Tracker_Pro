<objective>
Schließe den UAT-Gap `detail-modal-clutter` (CHAR-03, 06-HUMAN-UAT.md): Das Charakter-Detail-Modal (`showCharacterDetails`, features/party/party-details.js) wirkt "sehr unaufgeräumt", weil jeder der ~30 würfelbaren Einträge — 6 Attribut-Checks (.char-attr-box), 18 Fertigkeiten (.char-skill-item), 6 Rettungswürfe (.char-save-box) — ein DAUERHAFT sichtbares inline V/N-Buttonpaar (.char-adv-btns, ~60 farbige Buttons) trägt, obwohl die Zeile selbst bereits Klick-zum-Normalwurf ist (roll-char-*-stop).

Vom Nutzer gewählte Lösung: V/N nur bei Hover zeigen. Im Ruhezustand sind die V/N-Buttons versteckt; sie erscheinen beim Hover der Eltern-Zeile. Der Klick auf die Zeile = Normalwurf bleibt UNVERÄNDERT.

PFLICHT-Touch-Fallback: Touch-Geräte haben kein Hover. Wäre V/N dort nur per Hover erreichbar, würde Vorteil/Nachteil unerreichbar. Die Hide-until-Hover-Regeln werden daher in `@media (hover: hover) and (pointer: fine) { … }` gekapselt. Der DEFAULT (außerhalb dieser Media-Query) hält V/N sichtbar — Coarse-Pointer/Touch-Geräte behalten die Buttons.

Diese Änderung ist primär CSS. party-details.js benötigt voraussichtlich KEINE Änderung (die Hook-Klassen .char-adv-btns / .char-attr-box / .char-skill-item / .char-save-box existieren bereits) — nur anfassen, falls eine Hook-Klasse wirklich fehlt.

Purpose: Visuelle Dichte im Ruhezustand drastisch senken (von ~60 Buttons auf 0 sichtbar), ohne Funktionalität (Normalwurf, Vorteil/Nachteil) auf irgendeinem Gerätetyp zu verlieren.
Output: Hover-Reveal-CSS mit verpflichtendem Touch-Fallback in party.css, entschärftes Attribut-Grid-Spacing, und ein E2E-Case, der den Ruhezustand-Verstecken + Hover-Reveal beweist.
</objective>
